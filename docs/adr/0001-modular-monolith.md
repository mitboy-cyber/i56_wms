# ADR-0001: Modular Monolith with Event-Driven Core

## 状态
Accepted (2026-07-15)

## 背景
I56 Framework 需要同时支持多种业务产品（WMS/OMS/TMS/CRM/Finance），且需要能够演进为微服务架构。

## 决策
采用 **Modular Monolith** 作为 1.0 LTS 的架构模式：

- 每个业务模块是独立的 Go package
- 模块间通过 **EventBus** 通信，不直接调用
- 共享 Core 框架（认证/日志/缓存/配置）
- 模块内部使用 DDD Lite（聚合 + 仓储 + 服务）

## 替代方案

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 纯微服务 | 独立部署/扩展 | 运维复杂度高，不适合 1.0 启动 | ❌ |
| 传统三层单体 | 简单 | 模块紧耦合，无法拆分 | ❌ |
| **Modular Monolith** | 简单部署 + 可拆分 | 需要严格的模块边界 | ✅ |

## 模块边界契约
```
每个模块只能：
1. 订阅其他模块的 Event
2. 通过 Core API 调用共享服务

每个模块禁止：
3. 直接 import 其他模块的内部包
4. 直接访问其他模块的数据库表
```

## 后果
- 初期开发速度快（单进程部署）
- 未来拆分微服务成本低（事件已解耦）
- 需要开发团队严格遵守模块边界
